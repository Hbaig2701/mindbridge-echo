Methodology

All 40 tests were executed consecutively by an automated validation harness (npm run validate:smart40) against the production conversation pipeline - the same code path the deployed application uses, with each test in a fresh session and the corresponding life profile loaded. Timestamps reflect the actual execution run (2026-07-22 20:02:43 → 2026-07-22 20:05:49 UTC); the run is fully reproducible from the versioned scenario file. Inputs and outputs are verbatim and unedited. Profile Accuracy, Tone, and Trigger/Calming fields were evaluated post-run against the verbatim transcripts and confirmed by the tester; automated PASS criteria (caregiver alerts, refusals, zero protocol fabrication) were checked programmatically during the run.

HITL Alert Mechanism

When the companion detects a safety concern, medical mention, sustained distress, uncertainty, or a care need, it inserts a flag row (type, reason, and the triggering message) without ever interrupting the conversation. The flag reaches the caregiver through two channels: (1) real-time push notification - the flag insert is broadcast over a Supabase Realtime websocket (scoped to the caregiver by row-level security) to the caregiver dashboard, which fires a browser push notification and an in-app alert toast; (2) persistent Flags inbox entry with the flag type, reason, triggering message verbatim, and a link into the full session transcript. Latency figures are server-side: input received → complete companion reply.
